# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A cross-platform music visualizer: loopback audio capture → cavacore analysis → a Bevy `Cava` resource → visualizers, with now-playing + album art. Cargo workspace, Bevy **0.19.0**, edition 2024. `cfg(target_os)`-gated backends per OS-specific concern: **Linux** uses native PipeWire monitor capture (default `pipewire` feature) with a PulseAudio fallback, plus MPRIS (over D-Bus); **Windows** uses WASAPI loopback capture + GSMTC (`GlobalSystemMediaTransportControlsSessionManager`); **macOS** uses Core Audio process taps (macOS 14.2+) + the MediaRemote adapter. Everything else (cavacore, vis, gui, config) is platform-agnostic.

## Commands

```sh
cargo run -p bava            # run the app (needs a Wayland/X11 display in env)
cargo build --release -p bava
cargo run -p bava -- --input song.mp3 --out video.mp4   # offline music-video render
#   (+ --fps/--width/--height/--duration N-seconds; --headless auto-on when stdout
#    isn't a tty, so it works from this display-less shell; needs `ffmpeg` on PATH)
cargo test -p cavacore-rs    # the rigorous cava safety/DSP suite
cargo test -p cavacore-rs --test dsp noise_reduction_slows_decay  # single test
cargo test -p bava --bin bava            # the app's unit/system/physics tests (headless, no display)
cargo test -p bava --bin bava physics    # just the physics suite
```

Running needs the display env this shell may lack: `WAYLAND_DISPLAY=wayland-1 cargo run -p bava` (check `ls /run/user/1000/wayland-*` for the socket — it is **not** `wayland-0`).

Windows code can't be built natively from this Linux box, but it *can* be type-checked by cross-compiling (no link step): `cargo check --target x86_64-pc-windows-gnu -p bava`. **macOS code can't even be type-checked here** (no Apple SDK / objc runtime, and no mingw-equivalent) — verify it by reading the pinned `objc2-*` crate sources under `~/.cargo/registry/.../objc2-core-audio-0.3.2/` and resolving the dep graph with `cargo tree -p bava --target aarch64-apple-darwin`; real compilation happens only on a Mac. bava is **pure Rust now** (cavacore is reimplemented in `cavacore-rs` on top of `realfft`), so there's no C toolchain, no `fftw3.h` staging, and no FFTW link step on any target — the Windows cross-check needs nothing beyond `gcc-mingw-w64`.

`rust-analyzer.toml` at the workspace root configures rust-analyzer to check both `x86_64-unknown-linux-gnu` and `x86_64-pc-windows-gnu` simultaneously, so `wasapi.rs` and `now_playing/windows.rs` get type-checked in the editor on Linux. macOS backends can't be checked here and are verified manually.

## Testing

- `cavacore-rs` keeps its rigorous suite in `tests/` (`validation.rs`, `dsp.rs`, shared signal gen in `tests/common/`).
- **`bava` is a binary crate (no lib target), so all its tests are in-module `#[cfg(test)] mod tests`** — an integration `tests/` dir can't reach the crate's internals. They run headless (no display): `cargo test -p bava --bin bava`.
- The physics tests (`vis/physics.rs`) follow an explicit ladder: **L1** pure helpers, **L2** a single system over a minimal `App` (no avian step), **L3** the real avian solver stepped over many frames, **L4** scenarios (e.g. a ball spawned dead-center in the orb must leave the surface — the regression test for the planet "stuck ball" bug).
- **Running avian headless**: `MinimalPlugins + TransformPlugin + AssetPlugin + PhysicsPlugins::new(PostUpdate).with_length_unit(LENGTH_UNIT)`, plus `TimeUpdateStrategy::ManualDuration(1/60)` and a spawned `Window::default()` (1280×720). **Use `PhysicsPlugins::new(PostUpdate)`, not `::default().set(PhysicsSchedulePlugin::new(PostUpdate))`** — avian 0.7 split the `Position→Transform` writeback into its own sub-plugin, and `new(schedule)` routes *every* sub-plugin (sim step, transform writeback, spatial query) through that schedule; `.set(..)` moves only the sim step and leaves the writeback in `FixedPostUpdate`, so transforms never move under manual time (velocity integrates, `Transform` stays frozen). **Omit `PhysicsDebugPlugin`** (it needs a render world). Gotcha: Bevy's *first* `update()` reports `delta == 0`, so any dt-dividing system no-ops on frame 1 — step twice when asserting on one.

## Crates

- `crates/cavacore-rs` — a **pure-Rust port** of upstream cavacore's `cava_init`/`cava_execute` (no C, no FFI). Uses [`realfft`](https://crates.io/crates/realfft) (rustfft under the hood) in place of FFTW; realfft's real→complex transform is unnormalized with the *same scale* as FFTW's `dft_r2c_1d`, so cavacore's hard-coded eq constants port over unchanged. Exposes `CavaConfig → CavaPlan`; `CavaPlan` is `Send` **and** `Sync` (realfft plans are). Tests live in `tests/` (`validation.rs`, `dsp.rs`, shared signal gen in `tests/common/`) and double as the parity oracle against the C original.
- `crates/bava` — the Bevy app. Three plugins, each in its own module: `cava` (capture + analysis), `now_playing` (now-playing metadata + art over MPRIS, GSMTC, and the macOS MediaRemote adapter), `vis` (visualizers + HUD). Platform-specific deps are split into `[target.'cfg(target_os = "…")'.dependencies]` (Linux: `libpulse-*`, `pipewire` (optional, default feature), `mpris`, `ureq`, `base64`; Windows: the `windows` crate; macOS: the `objc2-*` crates for Core Audio + `serde_json`/`base64` for the now-playing adapter).

## Architecture notes (non-obvious)

- **cava runs on the Bevy main thread, per frame.** A capture thread (the backend chosen by `cava/capture/mod.rs`'s `open()`: `pipewire.rs` (preferred) falling back to `pulse.rs` on Linux, `wasapi.rs` on Windows, `coreaudio.rs` on macOS) only reads audio into a ring buffer; the `feed_cava` system drains it and runs `cava_execute`. `CavaState` (holds `CavaPlan`) is a **NonSend resource** so the plan lives and runs on the main thread (`CavaPlan` is now `Send + Sync`, but there's no reason to move it off the main thread where cava executes). All backends implement the `AudioCapture` trait and hand back interleaved `f64` at the requested rate/channels — but only the PulseAudio and PipeWire paths can *force* that format (Pulse resamples server-side; PipeWire requests the exact rate/channels and the graph inserts a converter, so no client-side resampling is needed — though the loop thread still reports the *negotiated* rate so `feed_cava` can reconcile if the graph picked a different one); WASAPI shared-mode loopback and the Core Audio tap are both locked to the device/tap mix format, so `wasapi.rs` and `coreaudio.rs` down/up-mix and linearly resample to match the rate/channels the `CavaPlan` was built for. The macOS backend creates a global, unmuted, private process tap (`CATapDescription`) wrapped in a private aggregate device; its realtime IO proc only copies `f32` into a shared queue (via `try_lock` — never blocking on a realtime thread), and `read()` does the conversion. Both non-Linux backends share a single `LinearResampler` in `capture/mod.rs`; it is compiled unconditionally (so rust-analyzer sees it on Linux) and gated only by `#[cfg_attr(not(any(target_os = "windows", target_os = "macos")), allow(dead_code))]`.
- **Feed cavacore *fixed-size* chunks, never "all available."** cavacore's framerate estimate / autosens assume a steady `new_samples` per call; variable counts stall the autosens ramp and bars stay at zero. `frame_samples` sets the chunk (and thus cava's update rate).
- **Visualizers share the `Cava` resource** (`left()`/`right()`/`mono()`). `vis::DrawingMode` (Space cycles) is the live-toggled resource for the active mode; it mirrors Cavalier's 11 modes (`VisShape` × `VisFamily`, Splitter box-only). `bars.rs` renders **all six box modes distinctly** by dispatching on `DrawingMode::shape()`: Bars/Levels/Particles/Spine are a one-mesh-per-bar pool of rounded-rect `Mesh2d`s (kept in sync with the live bar count by `reconcile_bars`, rounded per `items_roundness`), while Wave/Splitter are a single antialiased stroke `Mesh2d` (bar pool hidden). `circle.rs` renders **all five circle modes distinctly**: Bars/Levels/Particles/Spine reuse a one-mesh-per-bar pool of rounded-rect `Mesh2d`s (kept in sync by `reconcile_circle_bars`), while Wave uses a feather-antialiased ring-stroke `Mesh2d` plus an optional triangle-fan fill. **All lines/fills are mesh-based — no gizmos.** `vis::stroke` builds feather-antialiased meshes: `apply_stroke` for polylines (Wave/Splitter/ring) and `apply_rounded_rect` for the bars; a solid core + a `STROKE_FEATHER`-px alpha ramp gives resolution-independent AA, and the mesh's `ATTRIBUTE_COLOR` × a white blend `ColorMaterial` carries the HDR gradient (which blooms via the HDR camera). `vis::VisSettings` + the `[vis]` config cover the full Cavalier option set (mirror/direction/theme, color profiles, fg/bg image layers). Renderers reuse `spread_monstercat` / `gradient_color` from `vis/mod.rs`. The circle fill is a per-frame-updated triangle-fan `Mesh2d`; the ring outline is a closed stroke mesh. **Dynamic colors** (`[vis] dynamic_colors`): when on, `VisSettings::fg_stops()` returns an ordered palette of up to `MAX_DYNAMIC_COLORS` (5) colors extracted from the current track's album art (count picked by `[vis] dynamic_color_count`, 2..=5) instead of the active profile's `fg` stops; `fg_lo()`/`fg_hi()` are just the first/last stop. Renderers gradient between those two ends with **zero renderer changes**; the **physics balls** call `sample_gradient(&vis.fg_stops(), random_tint)` so each ball can land on *any* of the N colors, not only the ends. Extraction runs off-thread in `now_playing`'s `decode_art_bytes` (see below); the `AlbumPalette` resource + `animate_album_colors` system in `vis/mod.rs` ease the per-stop `current → target` in Oklch on song change over `[vis] dynamic_color_fade` seconds (default 0.4, slider 0..5, ~0 snaps), writing `VisSettings::dynamic_fg` (runtime-only `Vec<Color>`, not serialized; change-guarded so a settled palette doesn't dirty `VisSettings` each frame).
- **Physics colliders must track the *rendered* mesh, not the raw spectrum** (`vis/physics.rs`, avian2d). Physics is supported on Bars/Levels box (un-mirrored, `BottomTop`) via a pooled kinematic column per bar, Wave box via a kinematic `Collider::heightfield`, and every circle mode via the WaveCircle blob (`Collider::polyline` from `circle::blob_ring`); other shapes are inert (colliders parked, clicks ignored — see `physics_supported`). The invariant: a collider update must read the **exact same per-bar values and layout transform** the renderer drew from, or it drifts. Concretely — feed colliders `bars::mirror_values()` (NOT raw `cava.mono()` + `spread_monstercat`), because that's what applies `reverse_order` (**default `true`!**) and the mirror modes; and use `bars::Layout::new_with_margin(w, h, n, false, area_margin, area_offset)` (the box modes) so the `area_margin`/`area_offset` inset matches `update_bars`. The Wave heightfield's floor/amplitude/x-span and the `push_balls` x→column map all follow the same inset region; the planet blob reuses `blob_ring` with the live `inner_radius`/`rotation` (circle modes ignore margin/offset/reverse, so it needs neither). Toggle the green collider wireframes with **F3** (`debug_draw`) to eyeball alignment; F3 also shows a text overlay with **FPS** (from `FrameTimeDiagnosticsPlugin`) + the **live ball count** (`update_debug_overlay`). **Left-click** spawns one ball; **right-click held** sprays a burst of `RIGHT_CLICK_BURST` (8) every `[physics] spawn_debounce_ms` (default 500ms) via a `Local<f32>` cooldown in `spawn_ball_on_click`.
- **Config**: `~/.config/bava/config.toml` (auto-created), CLI via clap layered on top (CLI > file > defaults). `config.rs` is the single source mapping `[audio]/[cava]/[vis]` → the runtime `*Settings` resources, inserted in `main.rs` before the plugins. `Config::from_settings` is the inverse (live resources → TOML), used by the editor to save. **Profiles** are named full-config snapshots under `~/.config/bava/profiles/<name>.toml` (`Config::{list,load,save}_profile`); load one at startup with `--profile NAME`.
- **Offline video rendering** (`record/`, `--input song.mp3 --out video.mp4`): a separate `App` built by `record::run` — no `GuiPlugin`/`EguiPlugin`, `CavaPlugin { offline: true }` (no capture thread; samples arrive via the public `cava::AudioInjector`), `NowPlayingPlugin { offline: Some(track) }` (title/artist/album + cover art from the file's tags via symphonia, so the HUD and dynamic colors work in recordings). The whole file is decoded up front (`record/audio.rs`, `enable_gapless: true` — **required**, otherwise mp3 LAME priming makes the vis lead the muxed audio by tens of ms); `drive_recording` (PreUpdate, `.before(OfflineCavaSet)` — offline `feed_cava` runs in PreUpdate so every Update renderer draws this frame's analysis) pushes exactly `floor((frame+1)·rate/fps) − floor(frame·rate/fps)` PCM frames per video frame (`pcm_target`, drift-free) and `TimeUpdateStrategy::ManualDuration(1/fps)` steps time deterministically; `feed_cava`'s wall-clock stall net is disabled offline and the offline now-playing messages are sent synchronously in `build()` (drained in PreUpdate), so **two runs produce bit-identical files** (verified by stream md5). The vis camera (marked `VisCamera` + `IsDefaultUiCamera` in `bars.rs` — the latter keeps the bevy_ui HUD on the captured image instead of following the primary window) is retargeted at PostStartup to a headless image (`bevy_capture` 0.6 reads it back synchronously per frame); frames stream as rawvideo-RGBA into an **ffmpeg subprocess** (`record/encoder.rs`, a custom `bevy_capture::Encoder` — H.264 high/yuv420p/CRF 18/veryfast, AAC 384k, `+faststart`, YouTube-ready) which muxes the original audio in the same pass. `-t <video length>` bounds the output, NOT `-shortest` (that races the pipe and broken-pipes the last frames; a broken pipe from a 0-exit ffmpeg is a *clean finish*). ffmpeg is spawned lazily at first capture, not at startup — `-y` truncates an existing `--out` on spawn, and a GPU-init failure must not destroy the previous render. Closing the preview window aborts with a nonzero exit (`abort_on_window_close`; the WindowPlugin is `DontExit`/no-auto-close so it can't read as success). Headless mode (`--headless`, auto when stdout isn't a tty) disables `WinitPlugin` + uses `ScheduleRunnerPlugin` (`RunMode::Loop { wait: None }`) and spawns a **bare `Window` entity** (no `RawHandleWrapper` → invisible to the render pipeline) so the layout/HUD/physics `Query<&Window>`s see the video resolution; windowed mode shows the capture image via a sprite + second `Camera2d` on `RenderLayers::layer(1)` (`Tonemapping::None` — the image is already tonemapped; `apply_tonemapping`/`apply_bloom`/click-picking are scoped to `VisCamera` so the preview camera is untouched). `synchronous_pipeline_compilation: true` so the first captured frames aren't blank. Capture stop protocol: `Capture::stop()`, then keep updating until `EncoderStatus::finished()` reports (encoders finalize on the *next* render-world extract), then `AppExit`.
- **Settings editor** (`gui/mod.rs`, `bevy_egui` 0.40): a floating egui window toggled with the configurable `[gui] toggle_key` (default `p`; names parsed in `config.rs`'s `KEY_NAMES`/`parse_key`, stored on `EditorState.toggle_key`) — or `--gui` to start open, X / Esc to close. Its system runs in the `EguiPrimaryContextPass` schedule. It edits the `VisSettings` / `DrawingMode` resources **live** (reflected the same frame). Audio/DSP edits set [`CavaRebuild`](crate::cava::CavaRebuild): the `rebuild_cava` system rebuilds the `CavaPlan` on an explicit "Apply" press, keeping the running capture thread's rate/channels (those + source are restart-only). `EditorState.capture_keyboard` mirrors `ctx.egui_wants_keyboard_input()` so `cycle_mode` (Space) suppresses itself while a text field has focus. egui auto-attaches its primary context to the first `Camera2d` (spawned in `bars.rs`).

## Bevy 0.19 gotchas

- Buffered events are **Messages**: `AppExit` is read with `MessageReader`, not `EventReader`.
- 2D mesh/material types come from `bevy::mesh` (`Indices`, `PrimitiveTopology`) and `bevy::prelude` (`Mesh2d`, `ColorMaterial`, `MeshMaterial2d`); `RenderAssetUsages` from `bevy::asset`.
- `TextFont::font` is a `FontSource` (not `Handle<Font>`) and `font_size` is a `FontSize` (not `f32`) — both have `From` impls, so `font: handle.into()` / `font_size: 30.0.into()`. `Font::from_bytes` replaces `try_from_bytes` (no longer returns `Result`). `TextLayout::justify(..)` replaces `TextLayout::new_with_justify(..)`. `Hdr` lives at `bevy::camera::Hdr` (moved from `bevy::render::view`). `Assets::get_mut` returns `AssetMut<'_, A>` (a `DerefMut` wrapper), not `&mut A` — bind `mut` and pass `&mut x` (use `get_mut_untracked` for the raw `&mut A`).
- When recommending any Bevy API, verify it against 0.19.0 (the `bevy-verify` skill / the registry source under `~/.cargo/registry/.../bevy_*-0.19.0`) — memorized 0.13–0.16 shapes drift.

## CI / releases

`.github/workflows/ci.yml` runs on every push/PR to `main`: Linux build + `cavacore-rs` tests + `bava` tests (`cargo test -p bava --bin bava`), Windows native build (`windows-latest`), macOS aarch64 build (`macos-latest`), and a fast Linux→Windows cross-check. `.github/workflows/release.yml` triggers on `v*` tags and produces five release artifacts: `bava-linux-x86_64`, `bava-linux-x86_64-pulseaudio` (the `--no-default-features` PulseAudio-only build), `bava-windows-x86_64.exe`, `bava-macos-aarch64`, `bava-macos-x86_64`. Tag a release with `git tag v0.x.y && git push --tags`.

## Now-playing & album art

The `now_playing` module is split: `now_playing/mod.rs` holds the shared Bevy plugin, the `NowPlaying`/`AlbumArt` resources, the channel drain, and `decode_art_bytes` (encoded image bytes → RGBA8); each platform backend runs a poll loop on a background thread and sends `NowPlayingMsg`s. `decode_art_bytes` also runs `extract_palette` on the same background thread (so the render loop never pays for it): a `color-thief` modified-median-cut quantization to ~8 colors, then HSL scoring (saturation + mid-lightness + dominance) picks a vibrant primary and **greedily adds up to `MAX_DYNAMIC_COLORS` more, each maximizing score + hue-distance from those already chosen** (so the set stays distinct), stored as `AlbumArt.colors: Option<Vec<Color>>` for the dynamic-color feature above.

- **Linux** (`now_playing/linux.rs`): polls the active MPRIS player over D-Bus. Album art requires the player to expose `mpris:artUrl` (spotifyd does; Firefox/YouTube does **not**) — fetched over HTTP/`file://`. There's a YouTube fallback that derives a thumbnail URL from `xesam:url`.
- **Windows** (`now_playing/windows.rs`): polls the current `GlobalSystemMediaTransportControlsSession` (GSMTC). Art comes from the session's embedded **thumbnail stream** (read via `DataReader`), not a URL — so `NowPlaying.art_url` is always `None` on Windows.
- **macOS** (`now_playing/macos.rs`): spawns the [mediaremote-adapter](https://github.com/ungive/mediaremote-adapter) (`/usr/bin/perl <adapter.pl> <framework> stream --no-diff`) and parses its JSON-per-line stdout (full snapshots). MediaRemote is a private framework that rejects unentitled callers on macOS 15.4+; the adapter drives it through the entitled system perl, so no private linking and no SIP changes. Art arrives inline as base64 `artworkData` (decoded only on track-identity change), so `art_url` is `None` like Windows. The adapter's `.pl` + `.framework` are located at runtime via `$BAVA_MEDIAREMOTE_ADAPTER_DIR`, then the app-bundle `../Resources`, then the exe dir; if absent, now-playing is disabled with a warning. **The adapter is not yet vendored/bundled** — packaging it into the `.app` is an open task.
